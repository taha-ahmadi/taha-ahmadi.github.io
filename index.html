Development of Blockchain Explorer

Story, challenges and results

What was the problem?

Blockchain Explorer was an independent project separated from the Wallex core, and it was for inquiry into millions of users' financial transfer records. This project was my idea, and after numerous meetings and identifying trade-offs, we started developing this system.

We had millions of transactions and requests for deposits and withdrawals in Wallex that we stored in our primary database. After a while, based on business needs, we wanted to query all requests for deposits and withdrawals of fiat currency and cryptocurrency and check with the values recorded in the primary database.

The problem was that because cryptocurrency transactions were recorded with a delay through distributed nodes, we wanted to ensure the accuracy of the information recorded in our database. This work helped us to identify and troubleshoot unexpected problems that might occur from our own system or external third-party services that we used. Therefore, we started thinking together with the team members to design this system.
In the beginning, we didn't reach the desired result with the team members, so in my spare time, I started to independently develop a basic MVP system, which could query our bitcoin transactions as a cron job using bitcoin nodes.
After a week of development, I showed this initial MVP to the senior manager of our technical team in a meeting, and they liked the project very much and welcomed it, and as a result, it was decided to create a separate team to develop this product. as a result, we were able to turn my presented MVP into a real product.

Challenges

We had various challenges at the beginning of this project, such as:
1. We cover a large number of currencies in Wallex for trading, and we had to use separate third-party APIs to query each currency, and these APIs were not free, and we had to look for them to reduce the company's costs. We were looking for free solutions and solve technical problems of using several APIs.
2. Some requests to third-party APIs could fail for any reason; we had to provide a solution for the failed requests so we could inquire about the authenticity.
3. At Wallex, we were continuously growing and developing, so during our development, we might have covered more currencies for trading; so, we had to use an architecture and structure that would allow us to have a scalable and maintainable project to add new currencies.

Design & Development

Our project was supposed to develop as follows: In short, we needed several commands to check the accuracy of the available transfer information, which would start working in different time intervals using cron jobs.

We used different design patterns for our service architecture to have a clean, maintainable, and scalable structure to cover new currencies.
Each cryptocurrency used a separate dedicated API, so we had different clients for each currency.

Part of Client source code:

We used singleton, factory, factory method, and facade design patterns to have a cleaner and more scalable structure to cover new currencies.

We used value objects to integrate information and a specific structure for our data in the system.

Now our problem was troubleshooting communication problems with third-party APIs. To solve this problem, we used a Listener, and for any reason, our request would fail with the API; we would add a record in the failed_requests database and communicate with them again using another service.

Result

By using this project, we were able to identify and troubleshoot many unexpected problems. We were able to have a more stable system for the core of Wallex and make the best decision for the future changes of the project.

Triangular Arbitrage bot

.

Triangular Arbitrage is an exciting project for programmers who loves to do something different, and it gives you an excellent opportunity to understand many financial subjects. It has many challenges, So I created a project for this purpose. Out of interest and for better performance, I used Go to develop an arbitrage bot for CEFI exchanges and used Nodejs to communicate with DEFI exchanges. Doing this project helped me to gain a deep understanding of arbitrage performance with a focus on triangular Arbitrage. Also, by executing it, I got a relatively good profit from the financial markets of cryptocurrencies

Challenges and risks

In general, the challenges and risks of this project are divided into two groups.
1. One is around the calculation of Arbitrage because if you get the calculation wrong, you can end up in trouble.
2. Two is around the execution

1. Poor exchange APIs
Where and how to get information and update our data is very important
And it's very easy to lose significant percent in Arbitrage
2. latency
How long does it take for your code to calculate and run? How long does it take to get the updated price? The code needs to be efficient. Speed is very important, so code should be as fast as possible. Some robots have the servers co-located to where the exchange servers are to reduce latency so they can get information faster than anyone else. 3. Picnic (Human error)
Problems such as mistakes in calculation, strategy, and program bugs
4. Api Call limits
5. Conversion
It's easy to get the conversion wrong, dealing with bids and asks or dealing with DEFI and dealing with automated moneymakers(market-makers); 6. Slippage
the time it takes to execute, the price moves a bit, and you're cut out from an arbitrage oppronotiy
7. gas fees
8. Manual or automatic Arbitrage
9. Competition
there are triangular arbitrage bots that are cod-located that you can't compete with. 10. Domain like doing it in DEFI or CEFI or transferring the coin and mixture of both.

Part of Arbitrage command steps

1- We write a command for the price of the exchange and every market goes to fetch the prices every few seconds and store them in Redis.
2- We compare the prices to see if there is an arbitrage opportunity or not with the list of orders
3- We go and check the balances, for example, we can trade up to 60 thousand dollars
4- We find the maximum amount that we can order
5- We put an order that we have to put in the queue because it slows down the command.
